# setup-job — 定型作業を対話で job 化する

あなた（Claude）の仕事：**ユーザーとの対話だけ**で job を裏で生成・登録し、
即テストで試し、結果を見せ、指摘で直し、夜間運用に載せる。
ユーザーにファイル名・パス・コマンド・plist を見せない／触らせない。

## 心構え（重要）

- 相手は非エンジニアかもしれない。専門用語（job.md, frontmatter, launchctl, マーカー…）を**口に出さない**。
- **8割埋まったら対話を切り上げる**。全項目を完璧に聞き出そうとしない。
  「うざい」と感じる前に止める。足りない分は実行しながら直す前提。
- 1問ずつ重くならないように。相手が「もういいよ」の空気なら即テストに進む。

## ステップ 1：対話で5つを聞く（非エンジニア向けの言葉で）

以下を、自然な会話で埋める。専門語に翻訳せず、生活の言葉で聞く。

1. **仕様**：「その作業、後輩に初めて教えるつもりで、やる順番に説明してもらえますか？」
2. **お手本（既存パターン参照）**：「過去にこれの“お手本”になる成果物ってありますか？あれば見せてください。」
3. **迷ったらどう決める**：「途中で迷いそうな場面ってありますか？そのとき、いつもどう決めてます？」
4. **絶対やらないこと（禁止事項）**：「これだけは絶対にやらないで、っていうのはありますか？」
5. **完了と確認方法**：「どうなったら“できた”と言えますか？合ってるかは普段どうやって確かめてます？」

→ だいたい埋まったら「いったんこれで試してみますね」と言って次へ。

## ステップ 2：裏で job を生成する（ユーザーには見せない）

聞いた内容から `jobs/<job名>/job.md` を作る。`<job名>` は作業内容から英小文字ケバブで命名（例: `weekly-report`）。

- frontmatter は `enabled: false`（テストに通るまで夜間には載せない）。
- 本文は5項目テンプレ。聞けなかった項目は「（未確認・実行しながら詰める）」と書いて空欄にしない。

```markdown
---
name: <job名>
enabled: false
schedule: nightly
---

# <作業のタイトル>

**仕様**: <やることを手順で。曖昧表現を避ける>
**既存パターン参照**: <お手本のファイル/場所。無ければ「なし」>
**迷ったら**: <判断指針。無ければ「最も自然な解釈で進める」>
**禁止事項**: <絶対やらないこと。作業範囲外への変更は常に禁止と明記>
**完了条件**: <どうなれば完了か。結果の確認方法>
```

> 危険操作（外部送信・課金・本番デプロイ・破壊的削除）はコアの安全境界が
> 物理的に止める。job にそれを書かせない／書かれていても止まる。

## ステップ 3：即テストして結果を見せる（ユーザーは「試して」と言うだけ）

その job 1つだけを launchd 経由で即実行する（他の夜間 job は巻き込まない）:

```
bash core/lib/test_job.sh <job名>
```

- 前提：`install` 済みであること（未ロードなら、先に install スキルを案内）。
- 出てきた summary を**専門語を外して**要約して見せる：何をやったか／できたか
  （completed）／確認が要るか（needs_review）／安全境界で止まったか（safety_blocked）／
  実際に作られた成果物。
- 推測で「できました」と言わない。`runs/<date>-summary.md` と成果物の実体を見て報告する。

## ステップ 4：指摘で直す → また試す

ユーザーの「ここが違う」を聞いて `jobs/<job名>/job.md` を編集し、再度ステップ3。
2〜3回で十分。完璧主義にしない。

## ステップ 5：夜間運用に載せる

ユーザーが「これでいい」と言ったら、`jobs/<job名>/job.md` の `enabled` を `true` にする。
これで毎晩の実行対象に入る。最後に一言：「今夜から自動で走ります。朝に結果が残ります。」

## やらないこと

- ユーザーにコマンドやファイルを触らせる
- 5項目を全部埋めるまで対話を引き延ばす
- 危険操作を job に仕込む（そもそも安全境界で止まる）
